Contribute to PleaseFix

PleaseFix is built so developers of every level can contribute safely. Pick the track that fits you, make one small change, and grow from there.

← What PleaseFix is · the code lives at github.com/pleasefix-1/pleasefix

First, get it running (10 minutes)

The whole stack is one command — Django app, PostgreSQL+PostGIS, Redis, object storage, and Caddy. You never install Python, GDAL, or a database on your machine.

git clone https://github.com/pleasefix-1/pleasefix.git
cd pleasefix
cp .env.example .env
docker compose up --build          # → http://localhost:8000
docker compose exec app python manage.py seed_sample_data   # a couple of demo issues

On Windows, follow docs/ONBOARDING-WINDOWS.md (WSL2 + Docker, or a one-click VS Code Dev Container). Run the checks anytime with:

docker compose exec app sh -c "ruff check . && mypy . && pytest"
How the code is shaped. It's a modular monolith. The protected core (report lifecycle, geo queries, auth, dispatch framework) is maintainer-owned; contribution zones — channel adapters, client templates, config, translations, and everything below — are where you start and earn your way inward. CI applies formatting and types for you, so review is about ideas, not style.

Where things live

core/models.py
the domain: Issue, IssueUpdate, Flag, reporter secrets, managers.
core/views.py + templates/
the server-rendered pages (report, browse, detail).
api/v1.py
the public django-ninja API; schema is a reviewed artifact (api/openapi.json).
core/importers.py
import an issue from a social/web link (Reddit, X, OpenGraph).
core/abuse.py + docs/ABUSE.md
login-free anti-abuse: honeypot, throttles, flags.
docs/DESIGN.md, docs/WHY.md
architecture, decisions, and the motivation.

🎨 Frontend track

You'll work in: templates/, static/, core/forms.py. The app is server-rendered Django templates with HTMX for enhancement — the lowest-skill-floor path on purpose. Two rules: every user-facing string is wrapped in {% translate %}/{% blocktranslate %} (Bahasa Melayu + English are both first-class), and everything must still work with JavaScript off.

Your first change: a status badge colour

  1. Open http://localhost:8000/issues/ — you'll see the seeded issues with status pills.
  2. Find the .status rules in templates/issues/list.html and tweak the "fixed" colour.
  3. Reload — the dev server auto-reloads. Toggle your OS dark mode to check both themes.
  4. Add a BM string if you added any English text: docker compose exec app python manage.py makemessages -l ms, translate in locale/ms/LC_MESSAGES/django.po.

Ready for a real one? Good frontend starters: consolidating the inline CSS into static/css/site.css, or wiring the flag button through HTMX. See GOOD_FIRST_ISSUES.md.

⚙️ Backend track

You'll work in: core/models.py, core/views.py, api/v1.py, core/importers.py. Strict typing (mypy) and ORM-only (no raw SQL) are enforced in CI. Migrations ship in their own PR. Geo data is GeoDjango on PostGIS — points are PointField(geography=True).

Your first change: add a field to the API

  1. Open http://localhost:8000/api/v1/docs — the live API explorer. Hit GET /issues.
  2. In api/v1.py, add a field to IssueOut (e.g. status_display) and populate it in _issue_out.
  3. Regenerate the reviewed schema: docker compose exec app python manage.py export_openapi_schema — CI fails if api/openapi.json is stale.
  4. Add a line to a test in core/tests/ and run pytest.

Then try: the Reddit comment-permalink fix, the first Celery task (EXIF-strip photos), or a PostGIS nearby-issues query — all in GOOD_FIRST_ISSUES.md.

🤖 AI / integrations track

The product for you is the API. PleaseFix is designed so third-party clients, bots, and AI features are first-class — you don't need to touch the core to build on it. The OpenAPI schema is at api/openapi.json, live docs at /api/v1/docs.

Your first change: talk to the API

  1. With the stack running, list issues: curl http://localhost:8000/api/v1/issues.
  2. Write a short script that reads the JSON and prints each issue's reference code and location — drop it in a new examples/ directory.
  3. Submit a report through the form flow, then fetch it back by its public id: GET /api/v1/issues/<id>.

Bigger swings (all opt-in, none block the core path): LLM category suggestion via the Claude API, an outbound status-change webhook, or a generated SDK. When you build an LLM feature, keep it behind a setting and degrade gracefully when unconfigured — see GOOD_FIRST_ISSUES.md. Community clients get listed in the registry — recognition is the currency.

Opening the pull request

  1. Branch, make your change, keep ruff/mypy/pytest green.
  2. Commit with Conventional Commits (feat(api): …, fix(ui): …), one logical change per commit.
  3. Open the PR. CI runs the same checks; a maintainer reviews intent.
This onboarding map points at real files — if you find it out of date with the code, fixing it is itself a great first PR. PleaseFix is an independent, open civic platform (Apache-2.0).
